# 更快的合并导出与精简片段面板设计

日期：2026-04-06

## 背景

当前“合并导出”为了规避黑屏、后一段卡顿和音频丢失，采用了“两段式编码”：

1. 先从每个选区提取并转码出中间 `mp4` 片段
2. 再把这些中间片段做一次最终合并并重新编码

这个方案稳定性比最初的 `copy concat` 更好，但有两个明显问题：

- 性能偏慢，因为每个片段会经历一次编码，最终结果再编码一次
- 右侧片段面板视觉层级偏多，虽然保留了全部功能，但信息密度仍然较高

此外，当前片段重叠检测使用“发现重叠后直接弹回原位置”的方式，交互比较生硬，不符合精细剪辑时的手感预期。

## 目标

- 在不回退到不稳定 `copy concat` 的前提下，提高“合并导出”的速度
- 保持偏保画质、保体积的输出策略，不用极端快速预设换速度
- 右侧片段区保留全部功能，但通过更宽布局和更低视觉噪音降低拥挤感
- 将重叠处理改成“边界吸附”，允许片段无缝相邻但不允许实际重叠

## 非目标

- 不引入服务端 FFmpeg 或本机 FFmpeg 依赖
- 不新增“极速模式 / 高质量模式”等多套导出模式
- 不删除现有的音频导出、视频导出、预览、暂停、删除等片段操作
- 不改动左侧视频预览与波形图的整体结构

## 方案概览

本次采用单次最终编码的合并方案，并同步收敛右侧片段列表与选区重叠交互。

### 1. 合并导出：从“两次编码”改为“单次最终编码”

新的合并流程如下：

1. 将参与合并的源视频文件挂载到 FFmpeg WASM 文件系统
2. 按片段顺序，为每个片段在 filter graph 中直接构建：
   - 视频：`trim=start=...:end=...`, `setpts=PTS-STARTPTS`
   - 音频：`atrim=start=...:end=...`, `asetpts=PTS-STARTPTS`
3. 将所有片段流统一送入 `concat=n=片段数:v=1:a=1`
4. 只对最终输出做一次编码，直接生成目标文件

这样避免了：

- 中间片段生成、落盘、再读取带来的额外开销
- 中间 `mp4` 片段再拼接时的容器兼容性问题
- 两次编码带来的明显时间成本

### 2. 右侧片段区：更宽、更紧凑，但保留全部能力

面板将继续保留：

- 预览
- 暂停
- 音频导出
- 视频导出
- 删除
- 总时长展示
- 分段导出 / 合并导出

但会做以下简化：

- 右侧栏宽度略增加，让按钮更容易排布
- 每个片段卡片改为两层结构
  - 第一层：序号、来源视频名、当前视频标记、时间摘要
  - 第二层：左侧预览控制，右侧导出与删除控制
- 起点 / 终点 / 时长由三块独立信息框，收敛成一条时间摘要
- 卡片装饰、边框层级和垂直间距减少，降低视觉噪音
- 保持底部导出区固定且清晰

### 3. 重叠检测：从“回弹”改为“吸附”

新的规则如下：

- 片段之间仍然不允许重叠
- 片段允许紧挨着相邻片段边界放置
- 当用户拖动或缩放片段时，如果将发生重叠，不再强制恢复到原位置
- 系统改为根据当前操作类型执行边界吸附：

操作规则：

- 左边界缩放到前一个片段时：
  当前片段的 `start` 自动吸附到前一个片段的 `end`
- 右边界缩放到后一个片段时：
  当前片段的 `end` 自动吸附到后一个片段的 `start`
- 整段平移时：
  如果会撞到前后片段，则整段停在最近的合法位置，片段时长保持不变

## 详细设计

### 合并导出实现细节

主要调整文件：

- `js/export.js`

计划抽出以下职责：

- 构建输入挂载信息
- 构建单次导出的 filter graph
- 根据所有片段生成视频与音频 trim 节点
- 生成最终 concat 图
- 执行最终编码并导出

核心思路：

- 不再生成 `temp_merge_x.mp4`
- 最终命令中为每个参与片段的视频来源追加一个 `-i`
- 如果多个片段来自同一个文件，仍然可以重复引用同一输入文件对应的流索引，或按当前挂载结果建立映射
- filter graph 输出统一命名为 `[v]` 和 `[a]`
- 最终编码仍优先：
  - `libx264`
  - 偏保画质的 `crf`
  - `aac`
  - `+faststart`

性能收益来源：

- 去掉中间片段写入和读取
- 去掉一次额外整体编码

稳定性来源：

- 所有片段都在同一个 filter graph 内重建时间轴
- 不再依赖 mp4 片段级容器拼接

### 右侧片段面板实现细节

主要调整文件：

- `index.html`
- `js/waveform.js`

结构调整：

- 略微提升右侧栏在桌面布局中的占比
- 片段头部只保留核心信息：
  - 序号
  - 来源视频
  - 当前视频标记
  - 一行时间摘要
- 操作按钮按功能分区，但不做隐藏：
  - 预览 / 暂停
  - 音频 / 视频 / 删除

交互目标：

- 一眼能识别当前片段来自哪个视频
- 不需要滚动太多就能扫完多个片段
- 导出前能快速检查顺序和时长

### 重叠吸附实现细节

主要调整文件：

- `js/waveform.js`

需要新增或调整的能力：

- 在 `region-update-start` 时记录本次操作的初始状态
- 推断当前更接近哪种操作：
  - 左边界缩放
  - 右边界缩放
  - 整段平移
- 在 `region-updating` 阶段计算相邻片段边界
- 若将发生重叠，则直接将当前 region 修正到合法边界

边界判断策略：

- 仅考虑当前视频内其他片段
- 以排序后的前一个和后一个片段为约束边界
- 保持最小长度限制，避免吸附后片段长度变为 0 或过小

## 错误处理

### 合并导出

- 若没有任何片段，直接返回
- 若 FFmpeg 未加载完成，继续提示用户稍后重试
- 若 filter graph 执行失败，保留错误日志并中止下载
- 若主编码配置失败，可保留现有备用编码回退，但只作用于最终输出，而不是中间片段

### 片段交互

- 若吸附后结果无效，使用最近合法值而不是原地回弹
- 若相邻片段数据异常，降级为当前的合法边界裁剪逻辑

## 测试计划

### 自动化测试

新增或更新测试覆盖：

- 合并导出命令构建
  - 确认不再生成中间片段转码链路
  - 确认最终命令使用 `trim/atrim + concat`
  - 确认最终只编码一次输出
- 右侧片段面板结构
  - 确认卡片信息为摘要化结构
  - 确认所有操作按钮仍然存在
- 重叠吸附逻辑
  - 左边界吸附到前片段 `end`
  - 右边界吸附到后片段 `start`
  - 整段平移碰撞后保持时长并贴边停住

### 手工验证

重点回归以下场景：

- 两段来自同一视频的合并导出
- 两段来自不同视频的合并导出
- 输出视频是否仍存在黑屏、后一段卡顿、音频丢失
- 导出速度是否较当前实现明显改善
- 右侧面板在桌面宽度下是否更易浏览
- 片段拖动到边界时是否表现为自然吸附而非生硬弹回

## 实施顺序

1. 先改 `js/export.js`，将合并链路切换为单次最终编码
2. 补合并导出相关测试并验证通过
3. 调整右侧栏宽度和片段卡片结构
4. 实现重叠吸附逻辑
5. 补交互测试与整体回归

## 风险与取舍

### 风险

- 单次 filter graph 方案在多输入、多片段时命令会更长
- FFmpeg WASM 下 filter graph 的调试成本高于简单拼接
- 重叠吸附如果边界判断不准，容易出现“跳动”或“贴错边”

### 取舍

- 本次优先解决真实性能瓶颈，而不是引入更多模式开关
- 保留所有现有功能按钮，不靠删功能换整洁
- 交互以“可预期、可连续编辑”为第一优先级

## 预期结果

完成后，系统应达到以下效果：

- 合并导出比当前稳定版更快
- 仍然避免黑屏、后一段卡顿和只剩第一段声音的问题
- 右侧片段区更清爽、更好扫读，但功能不缩水
- 片段编辑时，重叠处理更像专业时间线工具的吸附，而不是机械回弹
