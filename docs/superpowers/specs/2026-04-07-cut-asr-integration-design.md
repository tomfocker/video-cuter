# Cut 与 FunASR HTTP 服务集成设计

日期：2026-04-07

## 背景

`cut` 当前已经具备浏览器端视频装载、波形框选、片段整理和导出能力，但语音识别链路仍然建立在旧的 WebSocket 协议假设上。与之对应，现有可用的识别服务已经切换为 HTTP 文件上传接口，并稳定运行在：

- `http://127.0.0.1:8000/healthz`
- `http://127.0.0.1:8000/v1/audio/transcriptions`

当前前端还存在三类问题：

- 转录入口仍尝试走 WebSocket 流式上传，导致服务不可用
- 页面脚本引用了多处并不存在的 DOM 节点，部分功能一触发就会报错
- 语音相关按钮仍被 `.pro-feature` 统一拦截，即使后端已就绪也无法正常使用

此外，字幕与文本选区逻辑仍按“一个 token 一段时间”的老数据结构工作，和新服务返回的 `segments`、`subtitle_segments`、`srt` 结构不匹配，造成字幕下载与“按文字选区生成剪辑片段”的体验都不理想。

## 目标

- 让 `cut` 直接对接现有 FunASR HTTP 服务，浏览器端上传音频并拿到识别结果
- 默认兼容当前唯一目标模型 Fun-ASR-Nano，并优先使用服务端已经优化好的 `subtitle_segments` 与 `srt`
- 补齐前端缺失控件，使服务配置、连接检测、字幕下载、识别状态都可用
- 优化按文字选区生成片段的逻辑，避免“一字一蹦”，尽量按更自然的字幕段或连续词组生成片段
- 保持现有视频/音频分段导出与合并导出能力不回退
- 为与现有 Docker 服务联调补充清晰文档

## 非目标

- 不为 `cut` 新增独立后端
- 不实现多模型切换
- 不改造现有 ASR Docker 服务的 API 结构
- 不强制要求用户配置 LLM，LLM 功能只在用户配置后启用

## 方案概览

### 1. 传输协议切换到 HTTP

前端不再建立 WebSocket 连接，而是执行：

1. 使用浏览器内 FFmpeg 从当前视频提取单声道 16k WAV 音频
2. 通过 `fetch + FormData` 上传到 `/v1/audio/transcriptions`
3. 指定 `response_format=verbose_json`
4. 将返回结果统一归一化为 `cut` 内部可消费的结构

健康检查不再依赖 WebSocket readyState，而改为调用 `/healthz`。

### 2. 统一转录结果结构

新增前端适配层，把服务返回的：

- `text`
- `segments`
- `subtitle_segments`
- `srt`

归一化为一个统一对象，至少包含：

- `text`
- `segments`
- `subtitleSegments`
- `displayChunks`
- `chunks`
- `srt`

其中：

- `displayChunks` 用于页面渲染和文本选择，优先使用 `subtitle_segments`
- `chunks` 保留兼容层，供旧的 LLM 与局部逻辑继续运行
- `srt` 优先保留服务端产物，下载时直接使用

### 3. 文字选区以“可读片段”为核心

页面渲染与鼠标选区不再只围绕单字 token。新的原则是：

- 显示时优先展示字幕级片段
- 用户拖选多个相邻片段时合并为更长的连续区间
- 生成波形选区时自动合并相邻或重叠的时间范围
- 排除模式仍保留，但基于连续片段而不是单字符

这样可以让“选一段话生成一个剪辑区间”更贴近真实剪辑需求。

### 4. 字幕下载优先使用后端结果

下载 SRT 时：

1. 如果转录结果自带 `srt`，直接下载
2. 如果没有 `srt`，再使用前端根据 `displayChunks` 兜底生成

这样可以直接继承后端已经做好的断句和时间轴优化，避免前端把字幕重新拆坏。

### 5. 页面交互补齐并解除伪 PRO 阻断

页面中补齐以下能力：

- 识别服务设置弹窗
- HTTP 连接检测状态
- SRT 下载按钮
- 双语字幕下载按钮
- 选区清空/重置/确认入口
- 当前选择模式与统计展示

`.pro-feature` 不再用于强制阻断真实可用功能，只保留视觉样式或完全移除。

### 6. 保持导出链路稳定

视频分段、视频合并、音频分段、音频合并继续沿用现有浏览器端 FFmpeg 方案。本次只修复它们与文字选区联动时的数据边界，避免因为转录结构变化导致导出失效。

## 详细设计

### 前端模块职责

- `js/websocket.js`
  - 保留文件名以减少入口改动，但内部职责切换为 HTTP transcription service
  - 负责健康检查、音频上传、结果归一化、状态更新
- `js/transcription.js`
  - 负责转录结果渲染、文本选择、选区合并、选区按钮显隐
- `js/utils.js`
  - 负责 SRT 时间格式化、后端 SRT 优先下载、前端兜底生成
- `js/main.js`
  - 负责事件绑定、设置弹窗、状态展示、去掉无条件的 PRO 拦截
- `index.html`
  - 补齐缺失 DOM 节点和服务配置入口
- `js/llm.js`
  - 改成消费统一后的 `displayChunks/chunks`

### 转录请求约定

请求字段：

- `file`: 提取后的 WAV 音频
- `model`: `fun_asr_nano`
- `response_format`: `verbose_json`

可选兼容：

- 若用户配置地址末尾带路径，前端会尝试规范化基础 URL，再拼接固定接口路径

### 页面状态策略

- 未加载视频：转录按钮禁用
- 已加载视频但服务未连通：按钮禁用，提示先检查服务
- 健康检查通过：按钮可用
- 正在转录：显示进度与日志，禁用重复触发
- 已有结果：允许下载字幕、做文本选择、触发 LLM 后处理

### 错误处理

- `/healthz` 失败时在设置弹窗和主界面都显示错误状态
- 上传失败时保留错误信息到结果区与底部日志
- 后端返回空文本时仍展示“未识别到语音内容”，不视为前端异常

## 测试策略

- Node 测试覆盖：
  - HTTP 健康检查状态文案
  - 后端返回结构归一化
  - SRT 下载优先使用后端 `srt`
  - 文本选区合并为连续时间段
  - 页面 HTML 中关键控件存在
- 手动验证覆盖：
  - 真实视频转录
  - 字幕下载
  - 文本选择生成剪辑区间
  - 视频分段导出与合并导出未回退

## 默认假设

由于本轮任务要求不中途等待确认，本设计默认采用以下约定直接实现：

- `cut` 与 ASR 服务在同机使用时默认地址为 `http://127.0.0.1:8000`
- 只兼容当前已验证可用的 Fun-ASR-Nano 服务
- 浏览器端继续承担音频提取与导出工作，不新增中间服务层
